Case Study 01 · Process Automation

Student Organization Fund Reconciliation Automation

BA Role — Analysis & Design

A real-world automation initiative for a student club's quarterly fund management workflow. The existing process required 2–3 hours of manual bank-statement cross-checking for 80 members and ~95 transactions. This project reduced reconciliation time to under 5 minutes through fuzzy name-matching and a simple dashboard to track payment status.

The Problem
Manual line-by-line reconciliation took 2–3 hours. Inconsistent bank transfer descriptions caused name-matching errors. No audit log existed, and unpaid members could only be identified after exhaustive manual effort.
The Solution
Auto-normalization of bank data, fuzzy name matching (≥80% threshold), automated payment classification, and a real-time dashboard with filter & export — shrinking reconciliation from 2–3h to <5 minutes.
AS-IS Process Mapping TO-BE Design User Stories Acceptance Criteria UI Mockup Fuzzy Matching Logic
Tested & Validated in Production

The application was deployed and tested against real bank statement data of the student organization. All reconciliation results matched the expected target: processing time reduced from 2–3 hours to under 5 minutes, with a match accuracy rate meeting the ≥80% threshold defined in the acceptance criteria.

Business Process Analysis — AS-IS vs TO-BE
AS-IS — Current State
  • Manual line-by-line bank statement checking
  • Name mismatch from inconsistent transfer descriptions
  • Manual Excel status updates after each match
  • No transaction audit log or history
  • Slow reporting — unpaid status visible only after full review
TO-BE — Optimized State

  • Clean up raw bank data automatically when imported
  • Match member names using fuzzy logic (≥80% confidence)
  • Sort payments into fund or fine categories
  • Flag uncertain matches (<80%) so they can be checked manually
  • Dashboard to filter, review, and export results
User Stories & Acceptance Criteria
User Story 01 — Upload & Reconcile
As a Fund Manager, I want to upload the bank statement and the dues report file, so that the system can reconcile member payments automatically.
Acceptance Criteria
Given Valid .xlsx files are uploaded, When clicking "Run Reconciliation", Then results are displayed within 30 seconds.
Given Match score ≥ 80%, When processing transactions, Then payment status is updated to Paid automatically.
Given Match score < 80%, When processing transactions, Then records are flagged as Needs Review for manual confirmation.
User Story 02 — Monitor Payment Status
As a Fund Manager, I want to monitor member payment status on a dashboard, so that I can identify unpaid or underpaid members quickly.
Acceptance Criteria
Given Reconciliation is completed, When opening Status Management, Then all members and their payment statuses are displayed in the table.
Given A keyword is entered in the search bar, When searching, Then matching members are filtered in real time.
Given The "Unpaid" filter is selected, When applied, Then only unpaid members are shown in the result list.

Case Study 02 · Jan 2026 – Apr 2026

E-commerce Platform for Near-Expiry Food

Team Leader & Business Analyst

A school project where we designed an e-commerce platform that helps stores sell near-expiry food to customers looking for discounted but still safe products. It includes location-based search, online ordering, and in-store pickup. We documented everything with UML diagrams and built a Figma prototype.

The Problem
Stores struggle to sell near-expiry inventory before expiration, leading to product waste, revenue loss, and low inventory turnover visibility.
The Solution
A platform with 3 main actors (Buyer, Seller, Admin + Payment Gateway) that lets users find near-expiry deals nearby, order online, and pick up in store. Everything was documented with UML and follows a 3-tier architecture.
